# Rust 贪吃蛇项目任务计划

## 目标

构建一个运行在终端中的 Rust 贪吃蛇游戏，优先保证：

- 核心玩法完整
- 输入响应稳定
- 终端渲染清晰
- 代码结构便于后续扩展

## 技术路线

- UI 方案：基于 TUI 终端界面实现
- 推荐框架：`ratatui`，不直接使用已停止维护的 `tui`
- 事件与终端控制：`crossterm`
- 随机生成：`rand`
- 错误处理：`anyhow` 或 `color-eyre`

## 库选型评估

### 1. `ratatui`

用途：

- 负责终端 UI 布局与渲染

结论：

- 适合本项目，建议作为核心 UI 库

原因：

- 是 `tui-rs` 的延续项目，生态更活跃
- 文档和示例相对完善
- 足够支持边框、文本、布局和自定义绘制

注意：

- 如果只做非常轻量的字符绘制，也可以完全不用 UI 框架，直接手动控制终端输出
- 但为了后续扩展分数栏、状态页、帮助信息，`ratatui` 更合适

### 2. `crossterm`

用途：

- 处理键盘输入
- 控制 raw mode、备用屏幕、光标隐藏等终端行为

结论：

- 基本必需，推荐使用

原因：

- 与 `ratatui` 配合成熟
- 跨平台支持较好

### 3. `rand`

用途：

- 随机生成食物位置

结论：

- 必需，轻量且够用

### 4. `anyhow` 或 `color-eyre`

用途：

- 统一错误处理

结论：

- 二选一即可

建议：

- 如果想先简单推进，用 `anyhow`
- 如果希望报错信息更友好、便于调试，用 `color-eyre`

### 5. 测试相关

可选库：

- 先使用 Rust 内置测试框架即可

结论：

- 第一版不需要额外测试库

### 6. 序列化与配置

可选库：

- `serde`
- `toml`

结论：

- 第一版暂时不需要

适用场景：

- 后续如果要保存最高分、键位配置、主题配置，再引入更合适

## 推荐的第一版依赖

```toml
[dependencies]
ratatui = "0.29"
crossterm = "0.28"
rand = "0.9"
anyhow = "1"
```

## 暂不建议引入的库

- `tui`：已停止维护
- 游戏引擎类框架：对当前终端项目偏重
- 配置和持久化库：第一版需求不足
